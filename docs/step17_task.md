# タスク: Step 17 本番稼働後のコードベース最適化とリファクタリング

## 目的
本番運用時のアドホックな修正によって低下したコードの可読性を改善し、技術的負債を解消する。同時に、現状の最新仕様に合わせてドキュメントとテストコードを同期させる。

## 実行要件

1. **可読性の向上とクリーンコード化**
   - 肥大化しているメソッド（特に `OddsAnomalyDetector` の判定処理や、`RaceOddsParser` のスクレイピング処理）を、単一責任の原則に従って適切な粒度のPrivateメソッドや別クラスへ抽出する。
   - 業務ドメイン（支持率、単複乖離、初期基準値など）の概念と、変数名・メソッド名が正確に一致するようリネームを行う。

2. **冗長・不要な処理の削除（デッドコードのパージ）**
   - アドホックなバグ修正時に残された不要なコメントアウト、過剰なデバッグ用 `System.out.println` またはログ出力、使用されていない変数・メソッドを完全に削除する。
   - 複数箇所で重複して記述されている計算ロジックや文字列処理があれば共通化する。

3. **ドキュメントと仕様の完全同期**
   - 現在の実際の動作仕様（URLの動的登録、動的スケジューリング、スプレッドシート永続化など）とドキュメントの乖離を解消する。
   - `README.md` を更新し、システムの全体アーキテクチャ、起動方法、各コンポーネントの役割を最新化する。
   - `CLAUDE.md` および `docs/skills.md` の実装ルールや型定義の前提を、現状のコードベースに合わせてブラッシュアップする。

4. **ビジネスロジックのテストカバレッジ拡充**
   - コアとなるドメインロジック（`OddsAnomalyDetector` の計算式、閾値の判定ロジックなど）において、カバレッジが不足している部分のJUnitテスト（正常系・異常系・境界値）を追加する。
   - 動的スケジューリング等の「時間に依存する処理」について、現在時刻をモックしてテスト可能なアーキテクチャになっているか確認し、必要に応じてリファクタリングした上でテストを記述する。

5. **【追加観点】インメモリ状態の管理と例外ハンドリングの強化**
   - `ConcurrentHashMap` 等でインメモリに保持しているアラート履歴や初期基準値について、メモリリークを防ぐためのライフサイクル管理（その日の全レース終了後、または日付が変わったタイミングでのデータクリア処理）が実装されているか点検し、不足があれば追加する。
   - スケジューラーのループ内での例外ハンドリングを再点検し、一部のデータ欠損や通信エラーがプロセス全体をクラッシュさせない堅牢性を担保する。
