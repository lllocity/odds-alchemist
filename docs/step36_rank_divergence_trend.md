# Step 36: 単複順位乖離の「変化方向」分析

## 概要

現在のロジックB（単複順位乖離検知）に「乖離の変化方向（拡大中/解消中）」の概念を追加する。乖離が発生している瞬間だけでなく、その歪みが進行中か収束中かを判別することで、より精度の高い馬券戦略の根拠を提供する。

---

## なぜ有用か（予想への活用観点）

### この検知が捉える市場現象

単勝と複勝のオッズには通常一定の相関がある（単勝人気1位の馬は複勝でも上位）。しかし、「3着には来るが1着は厳しい」という評価が市場で共有されると、複勝は売れるが単勝は売れず、両者のランキングに乖離が生じる。この乖離の方向性が予想の本質的な情報を持っている。

### 乖離拡大中の意味と活用

- **現象**: 時間が経つにつれて「単勝は売れないが複勝だけが売れ続けている」状態が強まっている
- **市場の解釈**: 関係者・玄人筋が「勝ちきれないが馬券には来る」という評価で複勝・ワイドを仕込んでいる可能性が高い
- **予想への活用**:
  - 複勝・ワイドの軸馬として積極的に採用
  - 三連複・馬連の相手候補として最優先
  - 拡大が続く限り「買い続ける」判断材料になる

### 乖離解消中の意味と活用

- **現象**: 単勝も売れてきて、乖離が縮小している
- **市場の解釈**: より多くの人が「1着もありうる」と評価しはじめた。本命視される過程にある
- **予想への活用**:
  - 複勝のオッズが下がりはじめているため、複勝単独での期待値は低下
  - 単勝・馬単での評価に切り替えるタイミング
  - 「乖離拡大中から解消中」という変化のトレースで、馬券購入の最適タイミングを計れる

### サイクル戦略

```
乖離発生（ロジックB検知） → 乖離拡大中（Step 36検知）→ 複勝・ワイドを仕込む
     ↓                          ↓
乖離解消中（Step 36検知） → 単勝・馬単に切り替え or 複勝を利確
```

### 他の検知パターンとの組み合わせ例

| 組み合わせ | 意味 |
|---|---|
| Step 36 + Step 34（フェーズ別） | 「直前フェーズで乖離拡大が始まった」→ 発走直前情報が複勝支持を後押しする非常に強いシグナル |
| Step 36 + Step 33（加速度） | 「複勝加速 + 乖離拡大」→ 複勝の急増と順位変動が同時確認。最強の複勝買いシグナル |
| Step 36 + Step 35（断層） | 「断層圏内の馬で乖離拡大」→ 勝負圏内かつ複勝重視の評価を受けている馬 |

---

## 目的

- 既存の順位乖離検知に「変化方向」のコンテキストを付加し、馬券戦略の精度を高める
- 「いつ買うか（拡大中）」「いつやめるか（解消中）」のタイミング情報を提供する

---

## データ構造の変更

### 追加するフィールド

```java
// 前回の乖離値: キー="URL:馬番", 値=乖離量（単勝順位 - 複勝順位）
private final ConcurrentHashMap<String, Integer> previousRankGap = new ConcurrentHashMap<>();
```

既存の `previousWinOdds`（Step 33 で `previousSnapshots` にリネーム）とは別フィールド。

---

## 計算式・アルゴリズム

### 乖離量の計算（既存ロジックBと同一）

```
currentGap = 単勝順位 - 複勝下限オッズ順位
```

> 値が正 = 単勝よりも複勝が上位人気（複勝が先行して売れている）
> 値が負 = 複勝よりも単勝が上位人気（通常は発生しにくいが検知対象外）

### 変化方向の判定

```java
Integer prevGap = previousRankGap.get(key);
if (prevGap != null) {
    int gapDelta = currentGap - prevGap;
    if (gapDelta > 0) {
        // 乖離拡大中: 複勝がさらに売れ続けている
        alerts.add(new AnomalyAlertDto(..., "順位乖離[拡大中]", gapDelta, ...));
    } else if (gapDelta < 0) {
        // 乖離解消中: 単勝が追いついてきた
        alerts.add(new AnomalyAlertDto(..., "順位乖離[解消中]", Math.abs(gapDelta), ...));
    }
    // gapDelta == 0 はアラートなし（変化なし）
}
// 前回乖離値を更新（乖離がある場合のみ保持）
if (currentGap >= RANK_GAP_THRESHOLD) {
    previousRankGap.put(key, currentGap);
} else {
    previousRankGap.remove(key);  // 乖離解消後は前回値をクリア
}
```

---

## 判定基準・閾値

```java
static final int RANK_GAP_THRESHOLD = 3;  // 既存の閾値を維持
```

変化方向のアラートは以下の条件をすべて満たす場合に生成する:

1. `currentGap >= RANK_GAP_THRESHOLD`（現在も乖離が存在する）
2. `previousRankGap.containsKey(key)`（前回乖離データが存在する）
3. `gapDelta != 0`（変化が発生している）

---

## 除外条件

1. 単勝上位3番人気は除外（既存ロジックBと同様）
2. 前回乖離データが存在しない場合（初回検知時）は変化方向のアラートを生成しない
3. 現在のギャップが閾値未満の場合、変化方向のアラートは生成しない（ただし `previousRankGap` から削除してリセット）
4. 複勝オッズが `null` または `<= 0` の場合はスキップ

---

## アラート形式

| フィールド | 値 |
|---|---|
| `alertType` | `"順位乖離[拡大中]"` または `"順位乖離[解消中]"` |
| `value` | `gapDelta` の絶対値（整数をdoubleとして格納） |

既存のロジックB（`alertType="順位乖離"`, `value=currentGap`）とは別のアラートとして生成される。両方のアラートが同時に発生しうる点に注意（乖離の存在 + その変化方向）。

**例（拡大中)**: `alertType="順位乖離[拡大中]"`, `value=2.0`（前回ギャップ3 → 今回5、delta=2）
**例（解消中）**: `alertType="順位乖離[解消中]"`, `value=1.0`（前回ギャップ4 → 今回3、delta=-1）

---

## 既存コードとの結合点

| 変更対象 | 変更内容 |
|---|---|
| `OddsAnomalyDetector.java` | `previousRankGap` フィールド追加 |
| `detectRankDivergence()` | 変化方向判定ロジックを末尾に追加 |
| `resetBaselineIfNewDay()` | `previousRankGap` のクリアも追加 |
| URL削除時 | `OddsTargetsController.deleteUrl()` で `previousRankGap` から当該URL起点のキーを削除 |

> URL削除時のキー削除はプレフィックス検索（`key.startsWith(url + ":")`）が必要。`ConcurrentHashMap` の `keySet().removeIf()` を使うこと。

## テスト方針

`OddsAnomalyDetectorTest` に以下のケースを追加する:

| テストケース | 期待結果 |
|---|---|
| ギャップ3 → ギャップ5に変化 | `"順位乖離[拡大中]"`, value=2 |
| ギャップ5 → ギャップ3に変化 | `"順位乖離[解消中]"`, value=2 |
| ギャップ3 → ギャップ3（変化なし） | アラートなし |
| 初回検知（前回データなし） | 変化方向アラートなし、次回以降のために記録 |
| ギャップ3 → ギャップ2（閾値以下に解消） | `previousRankGap` から削除、変化方向アラートなし |
| 上位3番人気のギャップ変化 | アラートなし（除外） |
| 日付変更後 | `previousRankGap` がリセットされ初回扱いに |

---

## 実装時の注意事項

- 既存のロジックB（`detectRankDivergence()`）は変更せず、その結果を受けて変化方向判定を行う構成にすること（単一責任の維持）
- `previousRankGap` の更新は「乖離あり・なし」の両方を考慮してクリアすること。乖離がなくなった後に古い前回値が残っていると、乖離が再発生した時に誤ったdeltaが計算される
- URL削除時のプレフィックス削除処理は `previousRankGap`・`previousSnapshots`・`previousCliffPosition` の3つに対して共通のクリーンアップメソッドとしてまとめることを推奨する（Step 33〜36の実装後にリファクタリング）
